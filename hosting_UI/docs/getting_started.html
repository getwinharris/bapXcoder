Getting Started with bapXcoder

Complete setup guide for the dual-model AI development environment with persistent project memory

Introduction to bapXcoder

bapXcoder is an advanced AI-powered Integrated Development Environment featuring a dual-model architecture with persistent project memory. The system combines an Interpreter model for communication and context management with a Developer model for specialized coding tasks, all connected via llama.cpp runtime to Hugging Face models without requiring local downloads.

Key Feature: Project-based memory system creates .bapXcoder directories with session continuity and todo management that persists across IDE restarts.

First Project Setup

When launching bapXcoder for the first time in a project directory, the system automatically:

  1. Creates a .bapXcoder directory for project memory
  2. Initializes user-specific memory in .bapXcoder/users/{user_id}/
  3. Sets up sessiontree.json for session continuity
  4. Creates todo.json for task and intent tracking
  5. Establishes the dual-model connection to Hugging Face

Understanding the Dual-Model System

bapXcoder uses a single agent with dual internal functions:

>

Handles all user communication, context management, multimodal processing (OCR, images), UI understanding, and project state awareness. This function mediates all interactions and coordinates with the Developer function when coding tasks are needed.

>

Specializes in coding tasks including code generation, refactoring, analysis, debugging, and implementation. This function only executes structured instructions received from the Interpreter function and never communicates directly with the user.

Model Connection Process

The system connects to Hugging Face models via llama.cpp runtime:

  • First access downloads model files to local cache (~5-6GB)
  • Subsequent accesses use cached models with faster startup
  • Uses allocated runtime quotas for model access
  • Secure connection to Qwen3-VL and Qwen3-Coder models

Basic Usage Patterns

Starting a New Task

Begin by establishing context with a detailed request:

"Analyze this project and help me implement a user profile management system. I need to update the user service to include profile picture upload and bio fields."

Adding Tasks to Memory

Use the todo system to track your intentions:

/Add: Implement user profile update endpoint - Priority: High /Add: Add profile validation middleware - Priority: Medium /Add: Write tests for profile functionality - Priority: Medium

Continuing Previous Work

When returning to a project, the system automatically restores your context and shows pending tasks.

Frequently Asked Questions

Q: Does bapXcoder require an internet connection?

A: Yes, bapXcoder connects to Hugging Face models via llama.cpp runtime. Project files stay local but model access requires internet.

Q: How does the 60-day trial work?

A: You get full access to all features for 60 days. After that, a subscription ($1/month or $100 lifetime) is required.

Q: Where is my project data stored?

A: Project memory is stored locally in .bapXcoder directories within each project, with user-specific subdirectories.

Q: What happens to my work if I don't subscribe after trial?

A: You can still access your project files, but AI features will be disabled until you obtain a license.

Q: How does session continuity work?

A: The system reads sessiontree.json and todo.json to restore your context, showing what you were working on and pending tasks.
