OAuth Integration

Secure Authentication Flow

User visits bapXcoder IDE | v User clicks GitHub/Google authentication | v Browser redirects to GitHub/Google OAuth | v User grants permission to bapXcoder app | v OAuth callback returns authorization code | v Server exchanges code for access token | v Server retrieves user identity (email, ID, name) | v User data stored encrypted in user's GitHub repo | v JWT token issued to user with encrypted subscription data | v User granted access to IDE features based on subscription

The authentication flow ensures:

  • Secure Identity Verification: GitHub/Google identities verified without access to repositories
  • Encrypted Subscription Storage: All subscription data encrypted in user's GitHub repositories
  • JWT Token Management: Secure tokens with automatic refresh capabilities
  • User-Specific Memory Spaces: Each authenticated user gets isolated session memory

Encrypted Data Storage in GitHub

Secure Data Architecture

All user data and subscription information is encrypted and stored in the user's GitHub repositories:

User authenticates with GitHub | v bapXcoder gets read/write access to user's GitHub | v Encrypted subscription data stored in: user's-github-repo/.bapXcoder/encrypted_data.json | v Data encrypted using user's authentication context | v During IDE operation: Encrypted access to subscription status | v Session data stored locally in .bapXcoder/users/{user_id}/ with user authentication

Data Encryption Details

  • Encryption Method: AES-256 encryption using user-specific keys derived from OAuth tokens
  • Storage Location: In user's own GitHub repositories for complete privacy control
  • Access Control: Only authenticated user can access their encrypted data
  • Local Mirroring: Local cache of encrypted data in project directories
  • Security Validation: All encrypted data validated before access

Subscription Continuity

Subscription State Across Sessions

The system maintains subscription continuity across IDE restarts:

On IDE Restart

  • Check if user JWT token still valid
  • Retrieve subscription status from encrypted GitHub storage
  • Verify subscription hasn't expired
  • Initialize appropriate access level based on subscription
  • Restore user-specific session memory
  • Display continuity message with subscription information

Subscription Expiration Handling

When subscription expires, the system:

  • Prompts for subscription renewal
  • Preserves all existing project memory and session data
  • Limits AI features but maintains IDE functionality
  • Allows immediate access restoration upon renewal
  • Maintains all session continuity and todo items after renewal

Frequently Asked Questions

Q: How is my subscription data stored securely?

A: All subscription data is encrypted in your own GitHub repositories, ensuring privacy and security.

Q: Do I need to log in every time I use bapXcoder?

A: Once authenticated, your session persists based on JWT token validity and subscription status.

Q: What happens to my data if I don't renew my subscription?

A: All project data and session memory remains preserved locally; only AI features are restricted until renewal.

Q: Can multiple users work on the same project with separate authentication?

A: Yes, each user maintains separate memory spaces in .bapXcoder/users/{user_id}/ with their authentication.

Q: How does authentication work with the dual-model architecture?

A: Both Interpreter and Developer functions respect user authentication for access control and memory isolation.
